# app_keepcon — 相对成熟长连接 / IoT 平台的待改进项

以下由「自建方案 vs 商用平台（如 AWS IoT Core、Azure IoT Hub、Ably/Pusher类、EMQX 企业能力等）」差距整理而来，**按重要程度排序**（高 → 低）。条目为演进 backlog；**当前产品阶段约定：首版只做 WebSocket，MQTT 进第二阶段**（见主计划），本表仍保留 MQTT/多协议相关项供后续对照。

---

## P0 — 可靠性与正确性（不做容易线上翻车）

1. **消息语义与投递保证**明确并逐步实现：**至少一次 / 至多一次 / 精确一次** 的边界；Redis Pub/Sub 的局限（无持久位点、易丢）与升级到 **Redis Streams / 队列 / 发件箱模式** 的触发条件。补：**死信、重试、幂等键** 的可观测与人工介入路径。

2. **顺序与分片约定**  
   同一 `device_id`（或会话）下的 **消息顺序**、跨 WS/MQTT 双栈时的一致性；多节点下 **禁止仅靠进程内路由**，Redis 路由 + TTL 与 **分片键** 文档化。

3. **发布与连接迁移（无损/低损升级）**  
   滚动发布时 **连接优雅摘除（drain）**、版本协商、尽量避免升级瞬间大量重复投递或丢 ack；与 C 侧车演进时的 **双写/读切换** 策略预案。

---

## P1 — 安全与多租户（企业场景硬需求）

4. **身份与凭证模型**  
   从「仅 token」扩展到可选 **X.509、JIT 注册、证书轮换**；MQTT 与 WSS 的 **ACL 一致语义**（租户 / device / topic前缀）。

5. **租户隔离与配额**  
   硬隔离（数据面 + 连接数 + 速率）与 **滥用防护**（边缘限流 + 业务限流）；审计日志字段与保留策略。

6. **合规与数据驻留（按需）**  
   区域部署、密钥管理、日志脱敏；与采购条款（等保、SOC2 等）对齐时的 **架构缺口清单**。

---

## P2 — 可观测性与运维（规模化必备）

7. **连接与消息 SLO指标**  
   连接数、在线时长、重连率、端到端延迟、按 **租户 / 版本 / 协议（WS|MQTT）** 切片；告警规则与仪表盘（可先在 `app_console` 或外部 Grafana）。

8. **分布式追踪与关联 ID**  
   从 HTTP 投递 → Redis → WS/MQTT 下发全链路 **trace_id / device_id / msg_id** 串联，便于排障。

9. **容量模型与弹性**  
   单机连接上限、Redis/DB 瓶颈、水平扩展公式；**自动伸缩** 策略与压测基线（定期回归）。

---

## P3 — 产品化与协议生态（增强竞争力）

10. **规则引擎 /  topic 路由（轻量即可）**  
    等价于云厂商「按 topic 转发到 HTTP/队列/存储」的最小能力，减少每个业务单独改网关。

11. **设备影子（Device Shadow）或期望状态模型（可选）**  
    与纯 inbox 消息互补，适合 IoT 配置下发与状态对账。

12. **与系统推送兜底联动**  
    APNs/FCM 与长连接 **离线策略**（合并、到达回执、与 inbox 状态一致），避免「只有长连接、无唤醒」的断层。

---

## P4 — 全球与体验（成本与复杂度最高，按需）

13. **多区域 / 边缘接入**  
    跨区域会话、数据复制、冲突解决；与 **单区域** 部署的明确分界（Non-goal 直到业务明确要求）。

14. **多语言 SDK 与接入文档**  
    降低客户端集成成本；协议版本化与兼容性矩阵。

---

## 说明

- **优先级**可按业务调整：若主要卖 B2B IoT，P1 会整体上移；若先做消息中台，P0 与 P2 更重。
