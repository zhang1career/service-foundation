# app_verify — TODO

与安全、风控、产品及运维相关的后续改进项（相对商用验证码/OTP 能力的缺口）。

## 1. 安全与风控

- **频控**：服务内未见按 IP、`ref_id`、手机号的限流/冷却。
- **错误信息**：错误统一成 `invalid or expired verify code` 等对调用方友好，但无尝试次数锁定、无渐进延迟。
- **绑定**：未见与手机号/邮箱的强绑定（只有调用方自管的 `ref_id`）；谁拿到 `access_key` 且能调 `/request`，就能拿到新码（取决于密钥保管与部署边界）。
- **传输**：若 `/request` 对终端或不可信网络暴露，明文回包等于把 OTP 暴露给同一客户端，和「只发到用户私有通道」的商用 OTP 模型不同。

## 2. 产品与运维

- **level**：有枚举并落库，但服务里没有对不同 level 的不同策略（长度、TTL、重试策略等）。
- **可观测性**：有 `verify_log`，但未见商用常见的控制台统计、告警、通道账单。
- **高可用**：依赖 MySQL；商用常配合 Redis 做短期存储与原子计数；此处未体现。
