# app_user — 与成熟用户中心 / Identity SaaS 的差距（待办方向）

本文档将 `app_user` 当前能力（账号、验证码流、JWT、资料与简单控制台等）与市面常见用户中心（如 Auth0、Okta、Keycloak、云厂商 IDaaS、Authing 等）对比，整理为后续可演进方向。**是否实现取决于产品定位**：仅服务内部微服务时许多项可刻意不做；若对标商用用户中心，下列为主要补齐面。

---

## 身份协议与联邦登录

- [ ] OAuth2 / OIDC 授权服务器形态（标准端点、client 配置、scope/claim）
- [ ] SAML 与企业 IdP 联邦
- [ ] 社交 / 第三方登录（按业务选型）

---

## 多因素认证与无密码

- [ ] MFA：TOTP、WebAuthn、短信等（按策略启用）
- [ ] 无密码 / 魔法链接等登录方式（若产品需要）
- [ ] 按风险或场景的步进认证（step-up）

---

## 令牌与会话治理

- [ ] Access/Refresh 策略：如 refresh 旋转、jti 与撤销列表（或等价服务端状态）
- [ ] 设备 / 会话列表、单设备登出、全局登出
- [ ] 敏感操作后失效旧令牌（如改密、风控触发）

---

## 组织、租户与权限

- [ ] 多租户 / 组织、部门、用户组（若多租户产品需要）
- [ ] 应用级角色与细粒度权限（RBAC/ABAC，按边界拆分）
- [ ] 邀请入组 / 入租户流程

---

## 用户生命周期与自动化

- [ ] SCIM 或等价目录同步（企业客户常见需求）
- [ ] 批量导入、规则化开户（与运营/IT 流程对齐）

---

## 安全运营与合规相关能力

- [ ] 登录与管理操作审计（可检索、可追溯）
- [ ] 限流、防刷、账户锁定与解锁策略
- [ ] 泄露密码库校验（如 Have I Been Pwned 类集成，按合规评估）
- [ ] IP / 地区等策略（按需）

---

## 隐私与合规产品化

- [ ] 同意记录与版本管理（若面向强合规场景）
- [ ] 数据主体导出 / 删除等产品化流程（与法务流程对齐）
- [ ] 与 `ext` 等扩展字段的边界：哪些进标准模型、哪些保留扩展

---

## 对应用与网关的标准契约

- [ ] `.well-known`、JWKS、token introspection 等（便于多语言客户端与 API 网关）
- [ ] 多应用（client）注册与凭证轮换

---

## 定时任务：过期未核销验证码与 event 对齐

- [ ] **背景**：`app_verify` 中校验码过期且未核销后，`sf_user.event` 上仍可能长期保持 `INIT` / `PENDING_VERIFY`，与真实可验证状态不一致，也不利于排查与统计。
- [ ] **方向**：增加定时任务（如 `django_crontab` 或与现有调度一致）：依据 `verify_code` 的 `expires_at` / `used_at`（及与 `event.verify_code_id` 的关联），将**已不可能再成功核销**且仍为待验证流程的 `event` 行 **`status` 置为 `9`（`FAILED`）**，`message` 可区分如 `expired` / `verify code expired`。
- [ ] **可选**：在同一或独立任务中对 `app_verify` 侧过期码做归档/删除策略（与留存、审计要求对齐后再定）。
- [ ] **注意**：任务应幂等、批量限流；避免误伤刚创建、尚未写入 `verify_code_id` 的极短窗口（若有则仅处理 `PENDING_VERIFY` 且 `verify_code_id > 0` 等条件）。

---

## 说明

- 当前 `app_user` 更贴近：**带验证码能力的账号与资料服务 + Bearer JWT**，与完整 Identity SaaS 不在同一产品层级属预期时无需全部跟进。
- 实施时建议按 **单职责与分层**（见仓库 `.cursorrules`）：协议层、会话层、目录/组织层分别演进，避免在一个模块内堆叠过多默认回退链。
