# app_config 相对成熟配置中心的待改进项

以下按 **优先级（P0 → P3）** 排列：P0 为安全与一致性基础，P1 为运维与可追溯，P2 为体验与规模化，P3 为平台化增强。实现 `app_config` 后可按序演进，不必首版全部完成。

## P0 — 安全与数据正确性

1. **敏感配置治理**：`value` 若含密钥、令牌，需明确策略——落库加密（含 KMS/字段级加密）、控制台脱敏展示、传输仅 HTTPS；避免与明文 KV 混用无规范。
2. **调用方鉴权与限流**：在 `access_key` 之外，按需增加 IP 允许列表、QPS/配额、异常告警，降低滥用与误调用对 DB/Redis 的冲击。
3. **配置校验**：对 `value` / `condition` 增加可选 **JSON Schema 或类型校验**（按 `config_key` 绑定 schema），在写入阶段失败，减少线上解析错误。

## P1 — 变更可追溯与回滚

4. **变更历史与审计流水**：持久化「谁、何时、改了哪条配置、旧值/新值摘要」；与查询响应中的 `audit` 互补（响应面向单次读取，流水面向治理）。
5. **版本与回滚**：为 `config_entry`（或发布单元）引入单调版本号或快照；控制台支持回滚到历史版本；与 `rid_gen` 缓存失效策略对齐。
6. **环境与命名空间（产品化）**：若用 `conditions` 模拟多环境，在控制台提供 **环境维度筛选、跨环境对比、批量复制**，降低误发 prod 风险。

## P2 — 客户端体验与实时性

7. **变更推送或 Watch**：在 HTTP 拉取之外，提供 **长轮询 / SSE / WebSocket** 等「配置变更通知」，减少客户端轮询与延迟；需定义与 `rid_gen` / 版本号的联动。
8. **多语言 SDK 约定**：文档化本地缓存、退避重试、与 `access_key` 的安全存储；可选提供参考客户端实现。
9. **灰度与圈选（进阶）**：在现有「条件匹配」之上，增加 **百分比、实例标识、名单** 等发布策略的可选一层（或明确由业务在 `conditions` 中自建约定）。

## P3 — 平台化与运维成熟度

10. **高可用与 SLO**：配置服务随 Django 部署时的多实例一致性、Redis/DB 故障降级策略、只读降级路径（若有）。
11. **备份与迁移**：`sf_config` 的备份恢复、跨环境迁移工具、大批量导入导出。
12. **审批与工作流（可选）**：敏感命名空间变更需双人复核或工单对接（视合规要求而定）。

---

**说明**：首版聚焦 P0 中与安全相关的条目及基础校验；P1 在业务强依赖「谁改了什么」时尽早做；P2/P3 按流量与组织规模迭代。
